# Diagnosis-First Positive KB Runtime Training

This file mirrors the runtime payload generated by
`backend/diagnosis_first_training.py`. It is an auditable prompt/training
surface for the webapp chain; it is not a new clinical rule engine and not a
case-specific diagnosis closer.

Core invariant:

```text
Correct diagnosis is the objective. Warnings and gates are safety boundaries,
not diagnostic success.
```

Batch A invariant:

```text
Use diagnosis-oriented KB cards only as reasoning aids. Do not activate a card
from a generic tag. Retrieval is not activation: the minimum positive anchor set
must be checked before a card is used as active support.
```

Runtime stage responsibilities:

- Evidence Reader / InitialReader: observe anchors only; do not close final
  diagnosis from positive overjet, lip protrusion, crowding, or surface
  convexity.
- Candidate Builder / KC: when cues conflict, generate ordinary convex or
  bimaxillary protrusion and compensated Class III / concave candidates.
- Differential Diagnoser / CM: explain why the tempting surface-protrusive path
  is plausible and why compensated concave fits better when positive anchors
  support it.
- Source/Subtype Integrator / SeniorClinician: derive source attribution and
  ShenGang subtype as structured/auditable meaning, with formal measurement
  boundary explicit.
- Treatment Mapper / SeniorClinician: map treatment from diagnosis, source, and
  subtype. For FSP19, MSE or skeletal expansion, maxillary protraction, and
  orthodontic-orthognathic boundary must remain visible.
- Safety Boundary / Critic if enabled: check contradictions after the diagnosis
  chain; do not replace diagnosis with warnings.
- Human Report: lead with current diagnosis direction, then evidence,
  misleading surface cues, source/subtype, uncertainty/next evidence, and
  treatment implication. Do not lead with gate jargon.

Batch A card-use workflow:

```text
case anchors
-> candidate KB retrieval by explicit card/context/card_id or specific anchors
-> minimum positive anchor check
-> differential contrast, including tempting wrong path
-> source/subtype integration with review boundaries
-> treatment implication from diagnosis/source/subtype/movement feasibility
-> uncertainty/evidence gaps
-> final answer or structured escalation
```

Runtime normalized fields expected:

```yaml
evidence_anchors:
card_retrieval_record:
candidate_diagnoses:
differential_contrast:
source_subtype_integration:
treatment_implication:
uncertainty_boundary:
safety_boundary_checks:
clinical_escalation_payload_if_any:
```

Card activation status must be one of:

```yaml
- active_supported
- candidate_needs_more_evidence
- negative_control_not_applicable
- no_card_match
```

Generic non-trigger controls:

```text
concave alone does not activate FSP19, ML29F, or CADDC.
protrusion alone does not activate ML29F or SPT-VSC.
asymmetry alone does not activate CADDC.
retreatment / extraction_history / missing_tooth / prior_treatment alone does
not activate RETX-EH.
triangular alone does not activate SPT-VSC.
false_protrusive alone does not activate TCFP16.
```

FSP19 positive training example:

```text
surface protrusion / positive overjet / crowding
-> possible dental compensation
-> compensated Class III / concave candidate
-> contrast against ordinary protrusive convex
-> upper-lower-source concave
-> lower triangular shallow concave
-> MSE + maxillary protraction + orthodontic-orthognathic boundary
-> formal ceph / Coben / occlusion / CR-CO confirmation boundary
```

Negative examples that must fail:

- Warning-only output: review warning without positive diagnosis.
- Wrong-treatment output: concave risk mentioned but ordinary protrusion
  extraction/retraction is the mainline.
- Surface-cue closure output: positive overjet + lip protrusion + crowding close
  ordinary convex or bimaxillary protrusion.
- Card retrieval without minimum positive anchor check.
- Generic tag activation.
- No-card case that outputs only "not enough KB" or warning language.
- Cross-card conflict that merges card names without differential comparison.

Audit boundary:

- Runtime payload includes the default projection of positive diagnosis card
  `fsp19_false_protrusive_compensated_concave`.
- The card is used as training example and anchor-check context only, not as a
  case-closing shortcut.
- Non-default card projections still require an explicit reason.
- Batch A cards remain L6 registry/helper/evaluator cards unless PM explicitly
  scopes L7 runtime integration.
- Old LLM-wiki/raw KB can create candidate backlog, but must not become fuzzy
  runtime retrieval before it has a diagnosis-oriented card draft and evaluator
  coverage.
